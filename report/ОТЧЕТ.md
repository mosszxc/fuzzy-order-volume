# 1. Анализ предметной области

## 1.1 Описание предметной области

Определение оптимального объема заказа у поставщика является одной из ключевых задач управления запасами в логистике и снабжении. Данная задача возникает в различных сферах деятельности: от розничной торговли до производственных предприятий.

### 1.1.1 Основные участники процесса

- **Закупщик** - лицо или отдел, ответственный за формирование заказов
- **Поставщик** - компания, предоставляющая товары или материалы
- **Склад** - место хранения товарных запасов
- **Потребители** - конечные получатели товаров (клиенты, производственные подразделения)

### 1.1.2 Ключевые понятия

**Объем заказа** - количество единиц товара, которое необходимо заказать у поставщика для обеспечения бесперебойной работы предприятия.

**Уровень запасов** - текущее количество товара на складе, выраженное в единицах измерения или днях потребления.

**Спрос** - ожидаемое количество товара, которое будет потреблено за определенный период времени.

**Срок доставки (lead time)** - время от момента размещения заказа до получения товара на склад.

**Стоимость единицы товара** - цена одной единицы товара у поставщика, которая может варьироваться в зависимости от объема заказа.

## 1.2 Проблематика задачи

### 1.2.1 Традиционные подходы

Классические методы определения объема заказа основаны на детерминированных моделях:

1. **Модель Уилсона (EOQ - Economic Order Quantity)** - определяет оптимальный объем заказа на основе фиксированных параметров:
   - Годовой спрос
   - Стоимость размещения заказа
   - Стоимость хранения единицы товара

2. **Модель с учетом дефицита** - допускает временное отсутствие товара на складе

3. **Модель с учетом скидок** - учитывает изменение цены при увеличении объема заказа

### 1.2.2 Ограничения классических моделей

Классические модели имеют существенные ограничения:

1. **Неопределенность параметров** - реальные значения спроса, сроков доставки и других параметров часто неизвестны точно
2. **Нечеткость критериев** - понятия "низкий спрос", "высокий уровень запасов" являются нечеткими
3. **Множественность факторов** - решение зависит от множества взаимосвязанных факторов, которые сложно формализовать
4. **Экспертные знания** - опытные закупщики принимают решения на основе интуиции и опыта, которые сложно формализовать

### 1.2.3 Преимущества нечеткой логики

Нечеткая логика позволяет:

1. **Учесть неопределенность** - работать с нечеткими понятиями типа "высокий спрос", "низкий уровень запасов"
2. **Использовать экспертные знания** - формализовать опыт специалистов в виде правил нечеткого вывода
3. **Учесть качественные факторы** - включить в модель факторы, которые сложно измерить количественно
4. **Гибкость системы** - легко адаптировать правила под конкретную ситуацию

## 1.3 Факторы, влияющие на объем заказа

### 1.3.1 Количественные факторы

1. **Текущий уровень запасов** (в единицах или днях потребления)
2. **Ожидаемый спрос** (единиц в месяц)
3. **Срок доставки** (дни)
4. **Стоимость единицы товара** (рублей)
5. **Минимальный объем заказа** (установленный поставщиком)

### 1.3.2 Качественные факторы

1. **Сезонность** - влияние сезонных колебаний спроса
2. **Надежность поставщика** - стабильность сроков доставки
3. **Срочность** - необходимость быстрого пополнения запасов
4. **Финансовые ограничения** - доступный бюджет на закупку
5. **Условия хранения** - ограничения по складским площадям

## 1.4 Области применения

Система определения объема заказа на основе нечеткой логики может применяться в:

1. **Розничной торговле** - определение объема закупок товаров для магазинов
2. **Производстве** - закупка сырья и материалов для производства
3. **Фармацевтике** - управление запасами лекарственных препаратов
4. **Пищевой промышленности** - закупка скоропортящихся продуктов
5. **IT-сфере** - закупка комплектующих и расходных материалов

## 1.5 Выводы по анализу

1. Задача определения объема заказа является актуальной и востребованной в различных отраслях
2. Классические детерминированные модели не всегда адекватно отражают реальную ситуацию
3. Нечеткая логика позволяет учесть неопределенность и экспертные знания
4. Система на основе нечеткой логики может быть эффективным инструментом поддержки принятия решений

# 2. Постановка задачи

## 2.1 Общая формулировка задачи

Разработать информационную систему для определения оптимального объема заказа у поставщика на основе алгоритмов нечеткой логики, которая будет учитывать нечеткие и неопределенные факторы, влияющие на принятие решения о размере заказа.

## 2.2 Цель разработки

Создать программную систему, которая:

1. Принимает на вход нечеткие параметры текущей ситуации (уровень запасов, ожидаемый спрос, срок доставки, стоимость)
2. Обрабатывает их с использованием аппарата нечеткой логики
3. Выдает рекомендацию по объему заказа в виде четкого числового значения
4. Позволяет учитывать экспертные знания и опыт специалистов

## 2.3 Задачи разработки

### 2.3.1 Аналитические задачи

1. Исследовать факторы, влияющие на объем заказа
2. Определить нечеткие переменные и их лингвистические значения
3. Разработать функции принадлежности для нечетких множеств
4. Сформировать базу правил нечеткого вывода

### 2.3.2 Проектные задачи

1. Разработать математическую модель на основе нечеткой логики
2. Определить структуру данных для хранения входных и выходных параметров
3. Спроектировать алгоритм нечеткого вывода
4. Создать модель предметной области в нотации IDEF0

### 2.3.3 Программные задачи

1. Реализовать систему фаззификации входных параметров
2. Реализовать механизм нечеткого вывода (Mamdani или Sugeno)
3. Реализовать дефаззификацию результата
4. Создать пользовательский интерфейс для ввода данных и вывода результатов

### 2.3.4 Тестовые задачи

1. Разработать не менее 3 контрольных примеров
2. Провести тестирование системы на различных входных данных
3. Проанализировать результаты и корректность работы системы

## 2.4 Входные данные

Система должна принимать следующие входные параметры:

1. **Текущий уровень запасов** (current_stock) - количество единиц товара на складе
   - Тип: вещественное число
   - Диапазон: 0 - 1000 единиц
   - Лингвистические значения: "низкий", "средний", "высокий"

2. **Ожидаемый спрос** (expected_demand) - прогнозируемое потребление за период
   - Тип: вещественное число
   - Диапазон: 0 - 500 единиц в месяц
   - Лингвистические значения: "низкий", "средний", "высокий"

3. **Срок доставки** (delivery_time) - время от заказа до получения товара
   - Тип: вещественное число
   - Диапазон: 1 - 30 дней
   - Лингвистические значения: "быстрый", "средний", "медленный"

4. **Стоимость единицы товара** (unit_price) - цена за единицу товара
   - Тип: вещественное число
   - Диапазон: 100 - 10000 рублей
   - Лингвистические значения: "низкая", "средняя", "высокая"

## 2.5 Выходные данные

Система должна выдавать:

1. **Рекомендуемый объем заказа** (order_volume) - количество единиц товара для заказа
   - Тип: вещественное число
   - Диапазон: 0 - 2000 единиц
   - Лингвистические значения: "очень маленький", "маленький", "средний", "большой", "очень большой"

2. **Степень уверенности** (confidence) - уровень уверенности в рекомендации (опционально)
   - Тип: вещественное число
   - Диапазон: 0.0 - 1.0

## 2.6 Ограничения и требования

### 2.6.1 Функциональные требования

1. Система должна работать с нечеткими входными данными
2. Система должна использовать базу правил нечеткого вывода
3. Система должна выдавать четкое числовое значение объема заказа
4. Система должна быть устойчива к некорректным входным данным

### 2.6.2 Нефункциональные требования

1. Язык программирования: Python 3.8+
2. Использование библиотеки scikit-fuzzy для работы с нечеткой логикой
3. Программа должна быть документирована комментариями
4. Программа должна быть протестирована на контрольных примерах

### 2.6.3 Ограничения

1. Система не учитывает скидки за объем (может быть расширена в будущем)
2. Система не учитывает ограничения по складским площадям
3. Система работает с одним типом товара (не мультипродуктовая)

## 2.7 Критерии успешности

Разработка считается успешной, если:

1. Система корректно обрабатывает входные данные в заданных диапазонах
2. Система выдает логичные рекомендации по объему заказа
3. Результаты работы системы соответствуют экспертным оценкам
4. Система успешно проходит все контрольные примеры
5. Программа полностью документирована и готова к использованию
